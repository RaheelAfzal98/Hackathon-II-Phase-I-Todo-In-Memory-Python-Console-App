## 1. Project Title

**The Evolution of Todo**
**Phase I: In-Memory Python Console Application**

---

## 2. Project Overview

This project simulates the real-world evolution of software systems. In Phase I, the goal is to design and build a **simple in-memory command-line Todo application** using **spec-driven development** and **AI-assisted coding**.

Students act as **Product Architects**, focusing on architecture, specifications, and development process rather than writing manual code. The project starts with a basic CLI script and is designed to evolve in later phases into a **distributed, cloud-native, AI-powered system**.

---

## 3. Project Goal

- Teach software evolution through incremental complexity
- Enforce **spec-first development**
- Use AI as the primary implementation engine
- Eliminate boilerplate and manual coding
- Evaluate students on thinking, planning, and iteration quality

---

## 4. Development Philosophy

- Manual code writing is **strictly prohibited**
- All development must follow the **Agentic Dev Stack workflow**:
  1. Write Specification
  2. Generate Plan
  3. Break into Tasks
  4. Implement via Claude Code
- **Claude Code** is the only entity allowed to generate or modify source code
- **Spec-Kit Plus** is the single source of truth
- Process quality is more important than feature count

---

## 5. Target Audience

- Students
- Hackathon participants
- Product architects in training
- Developers learning AI-driven development

---

## 6. Technology Stack

- Python **3.13+**
- **UV** package manager
- Command Line Interface (CLI)
- In-memory data storage
- Tools:
  - Spec-Kit Plus
  - Claude Code

---

## 7. Functional Requirements (Core Features)

The application **must implement all five basic Todo features**:

### 7.1 Add Task
- Create a new todo item
- Required fields:
  - Title
  - Description

### 7.2 View Task List
- Display all todo items
- Each task must show:
  - Unique ID
  - Title
  - Description
  - Completion status (Complete / Incomplete)

### 7.3 Update Task
- Modify an existing task using its ID
- Editable fields:
  - Title
  - Description

### 7.4 Delete Task
- Remove a task permanently using its ID

### 7.5 Mark Task as Complete
- Toggle task status:
  - Complete
  - Incomplete

---

## 8. Non-Functional Requirements

- Clean, modular, and maintainable architecture
- Clear separation of concerns:
  - Data models
  - Business logic / services
  - CLI interface
- Readable and idiomatic Python code (generated by Claude Code)
- Graceful error handling
- No premature optimization or unused features

---

## 9. Data Constraints

- All data must be stored **in memory only**
- No file system persistence
- No databases
- Data is lost when the application exits

---
